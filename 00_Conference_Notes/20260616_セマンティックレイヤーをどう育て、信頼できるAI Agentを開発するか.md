# 「作って終わりにしない タイミーのセマンティックレイヤー育成の現在地」株式会社タイミー / chanyouさん
- ここで言う、セマンティックレイヤー
  - データ基盤にある事前定義済みのテーブル
- 信頼を軸とした新規事業・プロダクト開発
  - スポットワーク⇒これまでに蓄積された”信頼”を、新たな事業・プロダクト開発に活用
  - 新事業：タイミーキャリアプラス
    - 就業支援
- 既存のセマンティックレイヤーの育て方
  - システムの破壊的変更がセマンティックレイヤーに影響を与える場合がある
  - 予期しない破壊的変更を防ぐための体制や運用プロセスを確立する
    - プロダクト開発チーム
      - PRレビュアーにアサインしてもらい、影響範囲と変更時期を把握する
    - セールスフォース管理チーム
      - カスタムオブジェクトの変更内容と～
    - よくリファクタリングで起こるので、急ぎではないので、延期交渉する事も
  - 事例：決算期変更の事例
    - 影響ある事は、誰からも教えてくれないので、社内外の変化から自分で影響があるかを判断して追従していく
  - 具体的なデータ分析のニーズを補足する
    - BIだと、分析できなかったら諦めるので意図がログに残らない
    - AIだと、ログが残るので、本来欲しかったニーズが分かるから、それを新たに作ったり
- そもそもセマンティックレイヤーを作るべきか？
  - コンセプトの検証フェーズ
  - スケーラビリティの検証フェーズ
- セマンティックレイヤーの立ち上げと育成のアプローチ
- Embedded SREのアプローチに学ぶ
  - SREの役割：プロダクトの信頼性の維持・向上
  - 中央のSREチームが多数の開発チームの相談を全て捌くのは、スケールしない
- 「セマンティックレイヤーを育てる」を分解してみる
  - １：ビジネスイベントと粒度を特定する
  - ２：ディメンションを特定する
  - ３：
- 新規事業の誰を巻き込むか
  - ドメインのエキスパート
- 取り組みの例
  - モブプログラミングによるデータモデリングの実践
  - 徐々に出来る事を増やしてもらう。過程で
    - 開発用ハンドブックの更新
    - skills作成
# 「Socrates × Looker　〜セマンティックレイヤーで進化するデータ分析エージェント〜」株式会社メルカリ / ucchi-さん
- スライド：Twitterに公開しています
- Socrates
  - メルカリ社内向けデータ分析AIエージェント
- Looker：セマンティックレイヤーの機能を備えたBI
  - アドホックな分析やダッシュボード作成ができる
- 課題
  - Socrates：誤ったSQLにより計算ロジックがずれる
  - Looker：手間がかかる
- SocratesｘLooker
  - Looker ダッシュボードを見る
  - そのダッシュボードを深堀する
    - Looker Exploreを使用
  - 仮説を深堀で、データを取得して確定していく
  - Slack Searchを用いる事も
  - 報告すべき深堀軸を決定
  - 最後に、Socrates Skillsに書かれた形式で結果を報告
    - 提案も
- Socratesは、デモでデータ集計ではなくて、データ分析を行った
  - データ分析を行うためには、品質の高いデータとコンテキストが必要になる
  - データ・コンテキストが必要
- データ
  - Looker APIを用いたデータ取得
    - 高品質なデータを取得できる
      - Text-to-SQLでは、クエリの柔軟性は高いがデータ品質が低い
  - 品質の高い順にデータを取得する
    - フォールバックが組まれている
  - データ品質とカバレッジを両立する
  - 中央のドメインとLookerを参照し分ける
    - 中央：全社のデータエンジニアが管理：重要指標
    - ドメイン：各地で管理
- コンテキスト
  - 大きく分けて2種類
    - 業界用語
      - Lookerで管理されている
    - 分析手順
      - Skillsにk際
      - Notion/slackからMCP参照
- 課題
  - 会社全体で見ると、まだスプレッドシートやスケジュールドクエリが主役
