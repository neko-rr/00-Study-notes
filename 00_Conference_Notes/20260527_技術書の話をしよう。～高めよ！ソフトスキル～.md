# 『ソフトウェア受託現場の「失敗」集めてみた。』	出石聡史
- コニカミノルタ㈱で働いていた
  - 色測定機器：ハード＆ニッチ
- 42の失敗事例
- 失敗か成功かの2択ではない
  - 沢山の失敗の上に、成功が乗っかってくる
    - 失敗の大きさと数で、会社が潰れてしまう
- 受託・受託開発の失敗
  - 自分史上最大の失敗が「受託開発」
    - 年単位での遅延、億単位での予算超過、
- ソフトウェア開発における問題の原因は、ほぼ「コミュニケーション」に起因する
- 願望しか書いていない「お気持ち仕様書」
  - 「こうしてほしい」を具体化せず、受託側に投げてしまう
  - 顧客の困り事も分からず、解決する仮説すらない
- 要望が消化不良を起こしている
  - 担当者の時間不足
  - 担当者の理解不足
    - 要望自体が分かりません
  - 担当者の知識不足
  - 担当者不足
- 委託側に要望を要求にしてもらう
  - 要望をどんな仕組みで実現するのか、決めるのは委託側の仕事
    - 要望や願望は、全て具体的な形に落としてもらう
- 海外だと、謎な物を作って来る「異世界仕様」
  - 世界に当たり前なんてない
    - 例：複数ファイル選択して、ドラッグアンドロップしたら1ファイルだけ
  - 書いていない事は、自由に想像して作る（一番楽な方法で作る）
- 委託側と受託側で見かけ上、ミッションが異なる
  - ソフトウェアを「なぜ」作るのかを共有していないと意味がない
# 『要件定義の極意』	松本均
- 元々、小さいSierに居た時の事業会社とのすれ違いの根本が要件定義と感じて執筆
- 要件定義は、世の中的にもこれが駄目だと全部すべっty多雨
  - DXプロジェクトの70%が失敗していると言われている
- 難しさ
  - ニーズをつかめない
    - 業務フロー図が表層止まり
  - 要件を描けない
    - 技術検証が困難
    - 必要な要件定義の規模が巨大化
  - 伝わらない
    - Biz⇔Tech翻訳が属人化
    - 増え続けるExcel等の資料作成と会議
  - 要件定義が要因で、プロジェクト全体が「機能不全」「予算超過」「遅延」となる
- 解決したい事に応じてRuleを組み合わせて、活用して頂けると嬉しいです
  - どういう人を対象か書いていないが
    - 20人くらいとか、他の部署が関わるものにアサインされた人
    - 2,3人くらいだと気合で何とかなる⇒調子乗る
  - ニーズを的確に把握
  - 要件を描く
    - 動くプロトタイプで誰もが触れる設計書にし（Rule4）、技術的な不確実性を検証で潰す（Rule8）。
    - 「作ってみないと分からない」を要件定義の段階で消す
  - 伝わる要件定義
    - 全成果物・全やり取りを透明化し、(rule14/15）、遠慮を排した本質議論を担保し（Rule16）、意思決定を1名に集約する（Rule18）。
    - 属認化した翻訳を仕組みで置き換える
- Q&A　書店で1章だけ読むならどれですか？
  - 1・18章です（笑）
