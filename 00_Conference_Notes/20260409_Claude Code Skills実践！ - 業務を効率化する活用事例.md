# SkillがSkillを生む：QA観点出しを自動化した	西村 憲悟 (@sontixyou)
- ルーティンワークに近い事を全部Skillsにしていこう
- 人間は、SKILL.mdを書かなくて良い
  - skill-creator-skillを使用するだけで、作成してくれる
    - 新しいスキルの作成
    - 既存スキルの修正・改善
    - スキルパフォーマンスの計測
  - プラグイン公開
- 話者の感覚では、「作成」だけだと、ちょっと冗長的だから、修正もさせる
- ファインディでもQA観点のリストの自動化をテスト中
  - 仕様把握⇒影響調査（既存機能へどんな影響）⇒観点出し（抜け漏れ怖くて、過剰に時間かかる）
  - 全ての作業をサブエージェントにして、まとめる
    - ワークフロー制御Skill⇒4つのサブエージェント⇒まとめSkill
- 良いSkillを作るためには「言語化」が大事
- Q&A一番作成して良かったSKILL
  - プルリクして、デフォルトのブランチに戻る際に、自動で戻るSKILL
# 機能・非機能の学びを一つに! Agent Skillsで月間レポート作成始めてみた	bun-san(@bun76235104)
- スキル配置の使い分け
  - ローカルプロジェクトスキル（Git管理）
  - ～/.claude（グローバル）
    - どこからでも使える汎用スキル
      - Pull Request作成
- 品質レポートSkill
  - 先月発見されたバグの情報（JIRA）
  - AWSコストの情報
  - 機能的なバグとインフラ以上の関連分析
- 前提条件・データ元
  - いつ・どこで・どんなバグが発生したか、JIRAチケットで管理していた
  - SREチーム中心に、インフラ観点のダッシュボードを作成してくれている
- やりたいこと
  - せっかくデータがあるから、統合して「品質」を広く捉えるレポートが欲しい
- 工程
  - １：APIでデータ取得
  - ２：playwright-cliでログ管理
  - ３：レポートまとめ
- クレデンシャルを持たせたくない時は、あえてパスワードを自分で入力するようにしている
- 効果
  - 月に1回レポートだが反響あり
  - あのバグとこのメトリクスの異常が繋がっているのか、視覚的に分かって良い
  - AWSコストの見直し
