# - 19:05	基調講演（30分）ウィル 氏「Conversation is the New Dashboard：属人性を排除する第4世代BIツールの勢力図」
- 属人性排除に力を入れた

  - セルフサービスのBIツールが市場を席捲して、今、AIネイティブなBIツールになる変革期
  - AI周りの更新が速い
- BIツール世代紹介
  - 1st（1990s）：トラディショナルBI　MicroStrategy,Cognos：キューブ構築・IT専任
  - 2nd（2010s）：セルフサービスBI　Tableau,Power BI：GUI操作・ビジネス普及
  - 3rd（2015～）：DWHネイティブBI　Looker,ThoughtSpot：コード管理・セマンティック層
  - 3.5（2020～）：セマンティック民主化BI　Steep,Lightdash：dbt直結・非技術者に解法
  - 4th（2024～）：AIネイティブBI　Lightdash?Codatum：AIからダッシュボードが作れる
- セルフサービスの限界
  - 突き詰めるとデータ人材を介さないと運用できない状態を卒業できるか？
    - メルカリとかだと、もうビジネス職が操作して判断できる
      - でも、ダッシュボードを作る時には、データ人材が介在している
- 第4世代のAIネイティブなBIツールとは？
  - AIが動くことを前提に設計されたBIツール
  - ユーザー体験はチャットUIの対話
  - グラフを作る作業すら不要
- 覇権を争う4つの陣営
  - 第2世代の刷新
    - Tableauでは、あらゆるフェーズでピンポイントで対話型で作業できるAI Agent機能を持っています。
    - 1年位前まで話者は、Tableauはコード管理が苦手と感じていた
      - dbt Semantic Layerをデータソースとして使える
        - ブラックボックスなXMLファイルの設定情報からの脱着しようと模索している感が伝わる
      - Selesforce社が注力しているAgentForceのAPIファーストの分析エクスペリエンスの目玉としてTablesu Next
  - 第3・3.5世代の進化
    - codatum：コダタム
      - 2023年設立：日本
      - 最低限が一通り揃っていて、コード管理もできる
      - 分析ドキュメントの管理もあり、バランスよく揃っている
        - NoteBook機能
          - マークダウンやSQLを埋め込める
          - YAMLも持っている
      - AIアシスタント機能
        - ダッシュボードやグラフを読み込んで、マークダウンで対話できる
      - TableauやLookerなどよりは安価
  - バイブコーディング
    - 完全内製の例
      - メリットと個人利用
        - 簡単に実装できて可視化も容易
        - 対話型とコード管理が実現できる
        - 個人がかき捨てで、グラフを使うケースだったらOK
      - 組織利用における課題
        - 組織レベルは難しい印象
        - 速度が遅い
    - squadbase
      - 2025年日本
        - アメリカ法人も出している
        - 最初から、バイブコーディングでダッシュボードを作成するために誕生したBIツール
        - 対話型でダッシュボードの修正指示をしやすいような予約語を活用している
        - Claudeのアーティファクトの強化版
          - 似たような見た目になっている
        - Excelくらいの数十万件のデータであれば、サクサク動く
        - ソフトウェア企業の大規模データニーズにも対応できるレベルになっている

        - AIが自発的に提案してくれるような世界を目指していて提案精度も向上している

# 講演②（20分）Taiki Hoshino 氏「dbt Semantic Layer × Steep を2年間運用してみて感じたこと」
- ㈱Gaudiyに1人目のデータ人材として入社
  - データチーム：当時、社員自分1名・業務委託1名の2人体制
  - 会社：エンタメ領域で「ファン国家」
  - 理想的なファンエコノミー
    - ファンの力で取ったアジア初のグラミー賞：BTSファン通称「ARMY」
- ファン国家を形づくるためのプロダクト群
  - MyAnimeList：価値伝播：世界最大級のアニメ・漫画コミュニティサイト
  - Gaudiy Fanlink：価値共創：コミュニティプラットフォーム
  - Financial Labs：価値分配：Web3時代のあたらしい金融インフラ
- データサイエンスチームが目指している世界
  - ファンひとりひとりの貢献を計測・可視化しうる社会シミュレーション
- Steep導入の背景
  - SQLの手間と負担
    - SQLを書くスキルを持っている人は多いものの、データ自体のドメイン知識が定着していない事が多く、以下のような状態だった
      - データチームに依頼してSQLを作成してもらう
        - データチームメンバーの工数負荷
      - Redashの既存クエリを修正し、データチームにレビュー依頼
        - SQLの作成・修正に時間がかかる
        - レビュー依頼の心理的負荷が大きかった
    - 新メンバーのオンボーディング課題
      - 新メンバー急増し、既存の数値に対してドリルダウンをすることで、事業の解像度を上げたいニーズが増えた
      - SQLを書いたこと無い人は、依頼するしかない
  - セマンティックレイヤーで解決を目指した
- dbt Semantic Layerを選んだ理由
  - dbt Cloudを利用していた事と、接続できるBIが他より多かったため採用
    - BIの乗り換えも考慮
- Steepを選んだ理由
  - 費用：最初、3人まで無料。1シート追加で費用
    - 導入する時に、使用してくれるか不安だった
    - 使用する人から順次解法できるから使い勝手良かった
    - モバイル版もあったから、経営層がタクシー内でダッシュボード確認したいという要望にすぐ応えられた
- 大変だったこと
  - スタースキーマの徹底が必要
    - 中間モデルをどう作成するべきかを意識しなければいけなかった
    - Steep上の制約で、SQLならばできるけれど出来ない事があり、めっちゃ考えた
      - 結果としてリネージが綺麗になったから、その後の開発が楽になった
  - アドホック分析は、残り続ける
    - 依頼数は減ったが、残り続けた
    - アドホック分析のためのBIは、別途必要
      - 動的なものに対応できなかったから、Redashを使用
    - 3ヶ月以上見られていないダッシュボードは、自動削除していくにしたから、カオスにはなっていない
  - プレビュー画面が無いから、破壊的改変しないようにめっちゃ頑張っていた
- 良かったこと
  - AIとの親和性
    - 意外と良いと言われている事と同じ事をしていた
- 定点観測
  - セマンティックレイヤーなど柔軟性のある仕組みで、事前定義されたメトリクスを安定的にモニタリング
- アドホック分析
  - AIの登場で、「どのBIツールを使うか」ではなく、「何を知りたいか」から逆算して分析手段を選ぶ時代へ
  - 実際、AIにSQL書かせて、使い捨てダッシュボードとして、7日日間だけWEBで見れるようにして捨てたりとか。
- 何を定点で見て、何をAIに問うか。使い分けが重要な時代
- Q&A dbセマンティックレイヤーの良い事・辛い事
  - 最初、使い方の違いが分からずに1年くらいドキュメント見て苦労した
    - メージャー～
