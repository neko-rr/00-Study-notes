# 事業の意思決定を変えるカスタマー行動データの設計（木内）
- 「席押さえ」リリース 
- ユーザー行動の解像度
  - 興味が3段階
    - どの店舗をタップしたか？
    - どの店舗を表示したか？
    - どの店舗を**認知**したか？
- 課題１：100秒で100ヒットすると、訪問は自動的に終了⇒短時間での大量ログ送信は不適合（スクロールするから一瞬）
  - 解決：リアルタイムパイプラインの構築
    - データエンジニアと協働し、独自のリアルタイムパイプラインを構築（この前にGoogleアナリティクスも使用していたが要件に満たさない）
- 課題2：ユーザー行動の文脈把握：ユーザーが見えている範囲は？見切れている？認知している？
  - 解決：アプリ開発スプリントに参画し、詳細な文脈情報をログに含める体制を構築
  - 可能な限り、アプリ側の判定情報を取得し、分析基盤の集計ボトルネックを解消
- 課題3：dbt microbatchモデルにおけるbackfill運用
  - 速さと正確さのバランス、実務と理想のバランス
  - 動的に使い分けて、事業への価値提供のスピードを第一優先した
- 課題4：microbatchでのステートフル処理
  - 日時跨ぎで訪問している客のデータが分断されている
# 企画からアナリティクスエンジニアへ：営業商談ツールの再構築（小野）
- 新卒2023年入社
- 飲食業域で営業社員が使用するツール「かなめ」
- 開発背景：Excelが9つ乱立（データダウンロードと加工も個人）
- 「かなめ」で全て集計確認を可能にした
- 商談準備工数40%（＝2.5万時間/年に相当）
- 利用率：90%
- 1つ目の課題
  - 企画担当者とアナリティクスエンジニアの「データの実態」に対する解像度の違いにより、仕様確認のラリーが大量に発生
  - 1つの確認だけで数日かかることもあり、開発スピードが低下
  - ⇒解決：企画者が自分でSQLで見た。。。
- 2つ目の課題
  - 直しにくく、拡張しにくい構成
  - 中間テーブルだけで100個以上ある
  - 指標定義と集計ロジックが中間層に混在しており、依存関係も複雑で、改修時の影響範囲が見えづらく、改修コストが増大
  - 解決：Mart層を再定義して、汎用的に使う情報をマートとして作成。固有は別の層に分離
    - マート層：データマネジメント（アナリティクスエンジニア）・データエンジニアリング（データエンジニア）
# 事業戦略実現に向けた協業とキャリアパス（登壇者全員）
- AEは事業やプロダクトの意思決定にどう関わる？
  - KPIは、0秒で分かるようにダッシュボード。深堀は、SQL。
  - あるある：ダッシュボード要求が高まりすぎて、保守だけかかって、見てくれない
    - 分析の柔軟性を意識するように変えた
  - 中長期を考えて、中身を変える
- 飲食領域におけるAEの面白さと難しさは？
  - 大量に選択肢がある中で組み合わせるのが楽しい
  - 難しさは、質の担保
  - 早くモニタリングやダッシュボードを定義を揃えて出すのが、大変だった
    - 営業の要望は、やってみた結果を早く知りたいという要望が強かった
- 速さが求められている：モニタリングチーム。モデリングチームとして、別チームが負債解消に動いて、それぞれがチャッキアップしやすいようにしている
- N1で見たいか、統計で見たいか。
