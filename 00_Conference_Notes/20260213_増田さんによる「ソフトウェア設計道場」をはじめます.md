# 目次
- 登壇者 増田亨(@masuda220) 氏
  - 競争優位を生み出す差別化戦略を効果的に実行するためのソフトウェア設計力を習得するための、相互に関係する4つの研修カテゴリを発表します。
- 1)【ビジネス理解】差別化戦略視点のビジネスモデリング
  - 目的・内容: 競争優位を生み出すための「差別化戦略」の視点から、ビジネス活動を理解し、開発すべき機能の優先順位を判断し、適切な技術選択をする力を養います。
- 2)【要件定義】差別化戦略の実行に焦点を合わせた要件定義
  - 目的・内容：差別化戦略の実行に焦点を合わせた要件定義を行うスキルを習得します。変動し不確実で複雑で多義的な状況で、協働創発的・探索的に業務要件、機能要件、非機能要件を定義する考え方とやり方を学びます。
- 3)【ソフトウェアエンジニアリング】差別化戦略を効果的に実行するためのソフトウェア設計
  - 目的・内容: 差別化戦略を効果的に実行するためのソフトウェア設計技法を学びます。競争優位性を生み出す中核の業務領域に焦点を合わせ、ソフトウェアを継続的に進化させる変更容易性に焦点を合わせたアジャイルなソフトウェア設計技法をコードレベルで学びます。
- 4)【データエンジニアリング】競争優位性を生み出すためのデータモデリング
  - 目的・内容: データ発生の多元化とデータ利用の多様性に対応し、差別化戦略を効果的に実行するためのデータ分析に焦点を合わせたデータモデリング技法を学びます。
# 講演
- 感想：この人が研修プログラムを作成した記念講演的なものみたい
- 自己紹介
  - 業務系アプリケーションの開発
  - 最近は、既存システムのぐちゃぐちゃを整理する手伝いをしている
- 現代のソフトウェア開発の課題
  - 2020年代から変わった
    - スマホが普及
    - クラウド
  - 変化し続ける複雑さ
  - 予測しづらい
- 従来
  - データ記録・データ転送・集計・レポート出力
- 現代
  - データを高度に活用した自動化・効率化・最適化
    - ソフトウェア開発の質とスピードが業績を左右
- ソフトウェアエンジニアの役割の変化
  - 事業活動の当事者（事業成果に責任を持つ一員）
  - 事業活動の自動化・効率化・最適化の推進役
  - ソフトウェアエンジニアの判断と行動が業績を左右する
    - 今までは、頼まれた事をやるだけ
- 変化し続ける複雑さ
  - 事業活動の変化と複雑さに直接向き合うことになった
  - 従来
    - システム化の対象範囲を限定し、要求を固定
  - 現代
    - 事業活動全体が対象範囲、変化し続ける要求
  - 多数のアプリケーション、大量かつ多種多様なデータ
  - 複雑な組み合わせ、利害関係の複雑さ、変化する複雑さ
    - 論理的に組み上げなきゃいけないが、論理だけで動かない調整が必要に
- 予測しづらい
  - 事業活動は「とりまく環境」に依存している
  - ソフトウェアシステムも「とりまく環境」に依存している
  - 外部環境の変化の方向、速度、振れ幅は予測しづらい
  - 外部環境の変化は、コントロールできない
    - 予測しづらい外部環境の変化に適切に対応する工夫が必要
      - 適切：時間が限られる、リソースが限られる
- モダンなソフトウェアエンジニア
  - 出典：SWEBOK V4
  - 知識体系
    - 全部を網羅している人は、いない
    - 地図として役に立つ
    - 実践には役に立たない（目の前の課題の助けにはならない）
      - 個別の状況では、実用的ではない
      - 一般化された全体像と一般化された語彙を覚えても現実世界のソフトウェア開発が上手くできるわけではない
  - 実践技法は、この「知識体系」の逆で、現場での効果が実証された経験則
    - 現実の課題に対する実用的で具体的な解法
- 現在的な課題に取り組む実践技法
  - この人が付けた名前：SCSD（戦略的協働創発型ソフトウェア設計）
    - ソフトウェア開発の主目的の変化に対応
      - 特に最適化
- SCSDの基本コンセプト
  - ソフトウェア設計を中心にソフトウェア開発を進める
    - この人の考え方：これが一番ありとあらゆるものに影響を与えている
  - 事業戦略の実現手段として、ソフトウェア開発に取り組む
  - 多様な知識・経験・技能を持ち寄って、協働して創発的にソフトウェア開発を進める
- ソフトウェア設計
  - 形の無い所に形を与え、今ある形を良い形に整える活動
    - 良い設計は、悪い設計より変更が楽で安全になる
- 事業戦略の実現手段
  - 競争優位を生み出す自社独自性（中核の業務領域）に焦点を合わせて、ソフトウェアを開発する
  - 他社と同じで良い一般の業務領域は簡略にすませる
    - 購入できるならば、購入する
    - 模倣できるならば、模倣する
  - 補完
    - CRUD/ETL機能簡易開発
- 事業戦略とソフトウェアの実装のつながり
  - 差別化戦略⇒ビジネスルール⇒業務ロジック
    - 競合他社と異なる自社独自の価値提案
    - 差別化戦略の実行手段
      - 適切な行動を刺激
      - 不適切な行動を制限
        - 例：高級ブランドでは、値引きをするとブランド棄損になるからしない
    - ビジネスルールに基づく、計算判断ロジックの実装
      - 自動化・効率化・最適化
        - 例：高級ブランドならば、値引き機能は不要。カスタマーサポートに特別な高貴さが必要みたいな
  - 事業戦略の実体
    - プレゼン資料や経営幹部のメッセージは、事業戦略の実体ではない
      - 実現できていない経営者の願望
    - 事業戦略の実体
      - 事業活動の現場の日々の判断と行動が集積したもの
        - 何も考えずに判断してやっている事が多い。だから、質問しても出てこない事も
      - 事業活動を観察し、分析することで、自社の優位性がどこにあり、そのために何をしているかを発見できる
- システム思考とモデリング
  - 複雑な対象をシステムとして捉えてみる
  - システム＝要素＋つながり＋目的
  - システムとは、複雑な対象そのものではなく、複雑な対象の特徴を抽象的に表現したモデル
  - 役に立つモデリング
    - 全体の中から要点を発見する
    - 重要な要素＋重要なつながり＋重要な目的
    - 要点を発見できると、小さな力で大きく持続的な成果を生み出す方法が見つかる
    - モデル（結果）ではなく、モデリング（活動）が重要
      - モデルが重要ではない。絵にかいた餅
      - 役に立つモデルを関係者で協働して創発する事で、関係者の判断と行動に方向性と一貫性が生まれる（腹落ちする）

    - 成果を上げるための秩序を生み出す
