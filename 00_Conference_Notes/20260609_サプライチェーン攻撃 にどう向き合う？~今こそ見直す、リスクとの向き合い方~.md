# Build・Ship・Runで考えるサプライチェーン防御 〜実行時(Runtime)から逆算するセキュリティ設計〜
- 見えない物は、止められない
- sysdig
- サプライチェーン攻撃とは？
  - 作る⇒届ける⇒動かす
  - どの段階も「信頼」の上に成り立っている
    - 信頼しているライブラリ
    - 信頼しているシステム
- 代表的な攻撃パターン：影響範囲
  - 悪意あるパッケージ混入：依存する全てのアプリ
  - もっと羅列あり
- 既存の防除では、未知はすり抜ける
  - 正規システムに似せたものは、すり抜ける
- なぜ今
  - AI
    - 人の目を通っていない本番環境へのビルドが増えた
- 攻撃は「自己増殖」フェーズへ
  - Shai－Hulud
  - GlassWorm
- 解決策
  - 1箇所で守り切ろうとしない
  - １：build：危険なものは作らない
    - CIでスキャンし、違反は止める
    - 中身を、SBOMで正確に把握する
  - 依存を「入れる前」に聞く設定
    - リリース直後を避ける
      - 1日/Yarn
      - 悪性版は、数時間～半日で除去される
        - 一泊置けば大半は踏まない
    - インストール時実行を止める
      - 攻撃の多くは、postin~
  - ２：ship：危険なものを通さない
    - レジストリのイメージも検査する
  - ３：Run：動いた攻撃を止める
    - 一番重要
    - 動いた瞬間の挙動は隠せない
      - 攻撃には、それなりにコマンドを叩く必要がある
      - ファイル操作
      - 権限
        - 誰が何になろうとしているか
      - 攻撃者は、だいたい同じ事をする
        - シェル実行⇒ファイル作成⇒外部通信⇒権限昇格
        - 細かく例を挙げている
      - Drift検知
        - イメージから変化したものを捕まえる
- 全件対応の時代は、終わった
  - 深刻度ではなく「実際に晒されているか」で絞り込む
  - 新しい脅威も”自環境で影響があるか”で分かる
- 対応は、ワークフローで自動化する
- 実践：3段階で始める
  - レベル１
    - まず見える化
# セキュリティの専門家じゃなくてもできる。「セキュリティ意識」をアップデートしてサプライチェーン攻撃への耐性を高めよう。	高塚 広貴さん (@tk3fftk )
- リンクが多いので、配布スライド参照
- リスクの定義
- サプライチェーン攻撃のリスクが漠然と不安になる理由
  - リスクの実現確率が読みにくい事
  - リスクが実現した際の被害の大きさが読みにくい事
- 知識を高めるのは難しいが、意識なら今から変えられる
  - 専門知識を高めるのは険しい道のり
  - 意識を高める
- 自分たちの現在地を把握する
  - 普段扱っている情報を理解・意識しておく
    - 本番環境のデータ
    - 開発環境のデータ
    - ローカルマシンにあるデータ
  - 脅威モデリングの考え方を取り入れる
- リスク管理
  - リスクは、発生するまで抽象的な概念にすぎない
  - リスク管理は、大抵の人が日常で実践している
    - 保険に入る、1本早い電車に乗る
  - 全てのリスクに対して対策をすることは、現実的ではない
    - 見えないふりをして影響がない事を祈るのも明らか違う
    - この2点の間の良い感じのポジションをとる
  - 具体と抽象の往復
    - 具体と抽象は別軸で視座を上げて考える
- 情報収集の習慣を付ける
  - リンク集
- 今すぐやる
  - 全部やったからといって、必ず守れるわけではない
  - 個人でやるよりは組織で、気合でやるよりは仕組で制限する
  - 公開から一定期間経過していないパッケージのinstallを防ぐ設定
  - GMO 無料
  - VScode系エディタの拡張棚卸し＋自動更新停止
  - GitHub Actionsで外部のActionを利用する場合、v1などのタグ指定ではなくSHAで固定する
    - 利用しているActionやReusable Workflowも
  - Docker Hardened 公式の最小構成
