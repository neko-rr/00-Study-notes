# 越境するAIのために、境界を取り払う 〜 AI時代の開発体験向上に向けたリポジトリ統合の取り組み〜	shibutani 氏	
- 元々、フロントエンドとバックエンドのリポジトリを統合（モノレポ化）した話
- バックエンド：Go
- フロントエンド：TypeScript
- 言語もエコシステムも違う
- AIによる変化
  - バックエンドもTypeScriptになってきた
    - フロントエンドと同じ言語・エコシステムを共有
    - 両方のリポジトリを管理するのが手間になってきた
  - AIがFE/BEを横断して開発する
    - 順々に⇒横断で高速に
- 当初は、多くの不安の声があった
  - 不安の中心は「遅くなる・重くなる」
    - 片方をpushすると、もう片方のCIも走る
    - リポジトリが大きくなり、IDE・Gitも重くなる
- 進め方：不安を、速さで打ち返す
  - CI改善
    - テスト実行範囲を絞る
      - mainとの差分から、変更パッケージとその影響範囲だけをテスト
      - Go：影響を受けるパッケージを逆引き
    - ビルドキャッシュを効かせる
      - go buildはGitなどのVCS情報をバイナリに埋め込む
    - CIでcheckoutを速くする
      - sparse-checkoutで不要なディレクトリをcloneしない
      - 履歴が要る場面は、blob-less cloneでサイズ削減
    - Go test CI
- 結果
  - FE/BEの一貫性
    - これまで、APIスキーマがズレる（最新の変更の取り込み忘れなど）
    - 共通ライブラリのメンテナンスが簡単に
  - 越境のしやすさ
    - これまで：スキーマ→BE→FEを別々のPRで行き来
    - 開発単位とPRの単位が一致しない
