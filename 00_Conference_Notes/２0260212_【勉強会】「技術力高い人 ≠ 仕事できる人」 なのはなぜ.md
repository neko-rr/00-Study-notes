# サマリ（技術力高い人 ≠ 仕事できる人なのはなぜ？）
- 入社1年目：エンジニア11か月目
  - 元未経験
  - 年間300冊読んでいる
- CTO・マネジメント：ヒアリング
  - ソフトスキルの土台があって、ハードスキル
  - ソフトスキルを磨く人が少ない
  - ソフトスキルが無いと、ハードスキル以前に勝負が付く
  - チームに入れると駄目とレッテルが付くと、剥がすのが大変
# 良いエンジニアの構成要素3点セット（それぞれの定義と伸ばし方は？センスではなく伸ばし方がある！）
- 地頭
  - ブレーキの使い分け
    -  コントロール不可能なものを悩んでも仕方ないから、使い方を学ぶ
    -  対症療法ではなくて、原因療法をしてほしい
  - DB：インプット量不足
  - CPU：処理速度の低さ
  - 帯域：他の考え事
  - 劣化：糖分の枯渇・睡眠不足
- コミュ
  - どっちかというと、マネジメントや事業責任者が重視
  - 減点を防ぐ
    - めんどくさがられない（＝相手の工数を使わない）
      - 工数：時間、認知や理解の負荷
      - 質問前にラリーを予測して伝える
  - 加点を狙う
    - 愛嬌：「この人になら良い情報を言ってあげよう・助言しよう・仕事回そう」と認知される
      - センスではなく、認知で太刀打ちできる
        - 例：ソーシャルスタイル理論
  - 仕事の授受だけでは2流（AIで十分）
    - 相手をやる気にさせれなければ、人間がやる意味が無い
- ゼンシン：前進ｘ全身
  - 重要なのは、メンタルの強さではなく、メンタルを復元し、前に進捗させること
  - 砕けない強さを磨くのではない
    - それは再現性が無いし、高負荷過ぎるリスクがある
    - そもそも手段であって目的ではない
  - 熱狂を持って、やり抜くマインドの事
    - どれだけの時間の長さと濃さを作れているか
      - 『人は、何かをやらない理由を見つけることにおいては、天才だ』
      - メンタルが落ち込んでいる時に重要な判断をしない（よくある方法論）
# 今からできるネクストアクション（学びは、行動変容に繋げてこそ価値がある！）
- 行動変容例：これで初めて、インプットがアウトプットになる
- アクションは「SMARTの法則」に設定する
  - Specfic：具体的
  - Measurable：測定可能
  - Achievable：達成可能
  - Relevant：関連性
  - Time-boun：期限の明確化
