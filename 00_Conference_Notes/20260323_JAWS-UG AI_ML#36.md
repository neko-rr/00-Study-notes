# FastMCP OAuth～ 伊賀裕展さん 株式会社エクサウィザーズ
- MCP認可　クライアント登録
- Pre-Registration
  - 既知の場合によくワークする
  - Cognitoでサポートされている
- 既知以外のクライアントでは？
  - DCR（）
    - Cognitoでサポートされていない
  - CIMD（Client Id Metadata Documents）
    - DCRと⇒が逆
    - Cognitoでサポートされていない
- FastMCP
  - MCP Serverの実装に使用できるフレームワーク
  - 高レベルフレームワークで少ないコードでMCP Serverを実装できる
- FastMCP Oauth Proxyでの認可
  - Oauth Proxyが入る事で、トークンが二重化される
  - 公式で図がある
  - 文字多いから資料見ること
- FastMCPとCognitoの統合
  - FastMCPでAWSCognitoProvaiderが提供されている
  - 簡単だよー
- Tokenの永続化とJWT Signing Keyの設定
  - 本番環境ではTokenの永続化が必要
    - client_storageを設定する
      - そのままでは、平文で保存される
  - ProxyがFastMCP Tokenの発行主体のため、Signing Keyもいる
    - jwt_signing_keyを設定する
# MCPサーバー、AWSの何処に置く？　荒牧さん AI/ML支部
- MCPには2つの方式がある
  - stdio：ローカルのOSの機能（プロセス間パイプ）を使って通信する方式
    - 基本、ローカルでしかできない 
  - Steramable HTTP
    - リモートでやるならば、一択
    - 仕様上絶対に要るのは、POSTだけ
      - 他は、必要ならば追加していけばよい
    - POST⇒JSON単発応答（パターンA）
      - 現在、多い
    - POST⇒SSEストリーム（パターンB）
    - GET⇒SSEストリーム（サーバー起点の通知：パターンC）
    - セッション管理（組み合わせ）
      - 下ほど、難しい
- サービス別の対応状況
  - MCP SDK互換性
    - Lambdaだけサーバーレスで変換ツール必要だから大変
