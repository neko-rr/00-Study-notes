# AIで職種の境界がなくなり仕事そのものが変わりつつある今PdMは何をすべきなのか？
- 組織フェーズ別
  - 周囲の専門性に応じて役割は変わる
  - 但し、作れる感覚を持って、検証しながら決められるPdMは、意思決定の質が違う
- デザイン思考から考える、プロダクトを成功に導く3つの視点
- ビジネスインパクトを判断するためのプロセス
  - １：ビジョン・大枠の方向性を決める
  - ２：ユーザーに本当に価値があるか？
    - 作れるか？ソース
- 作って決められるPdMが強い
  - 周囲の専門性に応じて役割は変わる
- 事例：新規事業立ち上げ
  - 早く出したい
    - これまでのプロセス：1年でリリース
    - 今回：3ヶ月でリリース
  - ユーザー検証を踏まえて、確度を高めた上でのリリース
    - PdMが画面のプロトタイプを作成したり
      - 職種越境でブラッシュアップする環境
      - 2週間でプロトタイプ作成
        - お客様仕様を満たしたアプリ
        - 今回は、お客さんが「仕様決まっている」と明言していたのでこの時間
        - 仕様決まってなくても、1ヶ月
      - 実際触れるプロダクトにより、お客様の意思決定の方法が変わる
        - ユーザーテストにおける検証制度が向上
        - テスト期間中にも開発を継続可能
- 課題
  - 関心が見た目に引っ張られる
    - 本来は、価値提供できているか？と言った要件・情報構造を議論すべき場面で、チームの関心が見た目に流れる
    - AIっぽさによって本質的な議論ができなくなる
      - 見る気が無くなる
      - 早くこのフェーズを抜けよう
  - 早くアウトプットが出せるが故、主観のアイデア出しに発展しがち
    - 偉い人がやってきて、主観で機能を欲しがる
    - 解決
      - いきなり作らない。前段を整理し合意する
      - 誰の課題をそもそも解決するのか議論
  - 個人のスピードが上がってもチームのスピードが上がるとは限らない
    - バイブコーディングでプロトタイプを作るというのは、デザインプロセスそのものを変える話
- AIをPdM業務に組み込み実践例
  - 価値検証に集中するためにAIっぽさを早く脱却するTIPS
    - 価値検証に集中するためにAIっぽさを早く脱却する
      - 口頭でのイメージは伝わらないので、画像を読み込ませる
      - AIっぽさとは？
        - 色・余白・タイポグラフィを調整する事で脱却できます
        - スライドのような見た目すっきりくらいまで行けば、議論に集中できる
      - 早く再現性を持って、作成するならDesign.mdファイルが有効
    - AIを活用したGA分析とイシューの発見
      - 事例分析
        - データ分析できる人が居ないから、何をテコ入れればわからないを解決
      - どんな分析をしたいかをAIと事前に壁打ちしてmdファイルに落としておく
    - AIを活用したUX/UI評価と改善案作成
      - UIの良し悪しを判断できる人がいないから、改善の優先順位を決められなかった
      - PdMが職種を越えて対応
        - UX/UIの評価
        - UXの分析
      - ホームページをPencilで再現
        - Cloud上で色々指示が可能
        - 無料で、フィグマみたいなもの
      - 3つのスキルでUXUIを評価し、改善レポートを作成。
        - スキルマーケットから取ってきた
      - 改善案を3つ並べて、比較可能（見た目）
- 人間が介在し、チェック・起動修正することで精度をあげる
# Q&A
- 基本的には、自分で判断せずにユーザーに判断してもらう事が重要
  - 意図した動きになっているか
  - 利用されているか
- ユーザーインタビューの価値検証は、変わりましたか？
  - やる事は、そんなに変わってない
  - AIで作成したから、質問や中身が変わるわけではない
  - ユーザーが触る環境がリアルに近くなってくる
- 人がやる価値
  - 何か違和感があるという時
    - アウトプットに違和感がある時、逆引きで仕様や要件等の資料を並べて、逆引きで探っていく
- プロトタイプをどのタイミングでどれくらいの塩梅で渡すか？
  - プロトタイプを当てる前にユーザーインタビューすると思うので、仮説である程度やったなと思ったら、当てに行けば良い
