# SaaSからAIへの過渡期の中で現在、組織内で起こっている変化	イオンネクスト株式会社 大久保 悟

  - 前面リプレイスではなく、コストと効率の「最適配置」がカギとなる
  - 既存SaaSが教えるのは「確率」のみ
    - ビジネスが真に求めるのは「次の打手」

- 独自シミュレーションモデル
  - 半年を要したシステムがわずか2週間で稼働
  - 開発期間のパラダイムシフト
  - データ主導のクロスセル
  - AIを組み込む事で既存CRMの限界を突破し、真のセールスインテリジェンス領域へ
  - 例：すぐには、仮説が分からないような繋がりを発見できている
# お客様の長期的興味の理解に向けて	株式会社メルカリ 田中 涼 （alpicola）氏
- メルカリのレコメンド画面比較
- 従来のUser to Itemレコメンデーション（keyword-based）
  - ユーザー行動履歴からキーワード抽出+関連キーワード
  - レコメンドは、メルカリにおける主要な購入経路の一つになっている
    - 短期的な行動履歴が反映され過ぎる（リターゲティング的）
    - 「検索履歴」と認知されているくらい直近の物がでてくる
- 新レコメンドモジュール
  - Goal：レコメンドがお客様のことをよく理解していて、自然と欲しいものが見つかると思ってもらえる状態
  - RetrievalとRankingの2-stage
    - 候補商品（10M+）⇒Retrieval（500～1000）⇒Ranking（10～20）⇒お客様と絞り込む
    - Retrieval：検索履歴を列で時系列にして、もっと広い次の14番目とか見るだろう予測を作成して学習している
      - 2Tower
        - User/item embeddingの類似度でマッチ度を表現
        - Vertex AI⇒Cloud bigtable←Rec API
    - Ranking
      - Residual MLPベース
      - Learning to rank
        - 購入＞いいね＞閲覧のみ、のような感じでより深いエンゲージメントを学習する
    - 新レコメンドモジュールのリリース
      - ABテスト後にリリース
      - 購入系指標の改善が見られた
      - 初期は、ranking stage無しだった。段階的改良の結果が上記
# Yahoo!ショッピングのレコメンデーション・システムにおけるML実践の一例 〜「候補選び（Retrieval）」から「並べ替え（Ranking）」までのバッチ処理パイプライン 〜	LINEヤフー株式会社 山下 直晃 氏
- Yahoo!ショッピングの商品との出会い方
  - 検索
  - ディレクトリー
  - レコメンデーション
- レコメンデーション
  - 受動的な情報の表示のため、待ち時間なく表示する必要がある
  - システムデザインの一般論
    - 数多くのアイテム
    - 高速にレスポンス
    - レコメンド・アイテム毎の順序づけ
      - 段階的に数を減らして順序集合を作成
- MLでレコメンド可能な商品集合の作成
  - 10^8（batch）⇒10^2（online）⇒10^1
- Batch
  - item-to-item＝基本のレコメンド
    - アイテム毎⇒Batch処理でおおむね完了
  - 高スループット（＝時間当たりの処理数）
  - ML
    - 似た物集めて
    - クリックや購買の傾向で順序付け
    - 計算環境：オンプレミス
      - Hadoop環境
      - Kubernetes環境
        - 分散処理
    - Scaling
      - カテゴリー毎（<25）の学習や推論
      - GBDTの学習・推論はYARN Spark上でSynapse ML lightGBM
- 基本的には、notebook + papermillでタスク実行
  - 実行結果でレポ―ディング
  - 開発と本番の実行環境が近い：エラーの再現・解決が簡単
- MLシステムの更新
  - 今まで5年++MLシステムを発展的に運用
  - 2回の大規模更新
  - PyTorch⇒TensorFlow
  - MLモデル＋システム
    - ハイパースケーラーやトップアカデミアの論文　VS　現場
      - リソースの非対称性：計算資源・人的資源
      - データの質と規模の乖離：実データVSベンチマーク
- 感想：キーワード多い。資料発表されることを祈ろう
