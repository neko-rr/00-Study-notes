# 『新規サービス開発におけるReact Nativeのリアル〜技術選定の裏側と実践的OSS活用〜』ファインディ株式会社	加藤 雄大さん

# 『少人数でもモバイルとWebを妥協なく開発する ── React Nativeの実践戦略』株式会社テラーノベル	東海 和豊さん
- モバイルアプリとWebの決定的な違い
  - モバイルアプリ⇒App Store/Google Playで配信
  - Webサービス⇒Googleの検索流入
- HTMLの返し方は2種類ある
  - CSR：クライアントサイド
  - SSR：サーバーサイド
  - テラードラマは、大量のコンテンツｘ検索流入⇒SSRが不可欠
- クロスプラットフォームのWeb対応
  - Flutter Web・react native web比較
    - どちらもSSR（完成HTMLを返す）が出来ない⇒Google検索対応が不十分
- React nativeとNext.jsと組み合わせる
  - 同じReactベース
- UIライブラリ：Tamagui
  - Webのパフォーマンスが特に高い
- ディレクトリ構成
  - モバイルとWebで同じUIを使用
  - apps/
    - next
    - expo
  - なぜReact NativeのコードでWebが動くか説明あり
- TypeScriptで統一した成果
  - 約7割のコードをWeb/モバイルで共有
  - Webのエコシステムをそのまま活用
  - Hot Reloadで高速開発
- 共通化の現実：1年で見えてきたこと
  - 初期成功例：packages/appに画面を実装
    - 両方から参照
  - 【1年後】WEB・アプリどちらかだけ反映させたい事が増えてきた
    - 共通化維持
      - UIコンポーネント
      - ビジネスロジック
      - 型定義・ユーティリティ
    - 分離検討
      - 画面レベルの内容
- Storybook x Figma MCPでUI実装を自動化
  - Figmaのデザインから実装・検証までをAIが一貫して実行
- Rozenite & agent-device
  - Agentに「目と手」を与える
  - Rozenite：エラーログの自動取得・修正、パフォーマンス分析等
  - agent-device：スクショ
- Q&A　ユニットテストどうしていますか？
  - 最近は、AIの活用でやりやすくなった
# 『少数チームでエピックと向き合う技術選択 ── React Native × Claude Codeの現在地』Ubie株式会社	坂本 一弥さん
- Ubieのプロダクトエンジニアは、エピックに必要な事を全部触る
- エピック：目標を達成する施策の単位
  - エンジニア1人～2人の少数メンバーで1つのエピックを担当する事が多い
    - 対応範囲が広くなる
  - コードを書くだけじゃない
    - 仮説検証サイクルをどれだけ速く回せるか
- なぜ少数チームで可能か？
  - アプリは基本TypeScriptで統一されているため、Web出身者でもアプリの開発がしやすい
- AIエージェントの普及で、職種間を飛び出して横断して染み出している
- 開発・デザイン・企画
    - 例：Devinを使用して、マーケがPR出したり
- UbieのアプリHome画面
  - 小さい機能が多い（ミニアプリの集合体）
    - 全部が同じ実装方法でもなく、Next.jsやネイティブ等使い分け
- WebViewの中からネイティブの機能を利用したい
  - アプリなら直接呼べる
  - Webからは呼べない
  - 公式で用意されていない
    - リクエスト⇒レスポンスの仕組が無い
  - RPCでネイティブを普通に呼べるようになる
- チームの共通言語としてのClaude Code
- ボトルネック
  - デプロイは、解決（OTA）
  - 分析⇒示唆⇒スキルで民主化
    - 文脈をスキルに埋め込む
      - データ層
      - 分析層
      - もう1個
    - スキルは、育つ
      - 今、48スキルがある
        - 分析・計測だけで17個
