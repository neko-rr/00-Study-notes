# 内容
- ログ読んでますか？
  - 多くの方は、自分が関わるログしか読まないのでは？
- ログの役割
  - 5W1H視点で出力されている事が多い
  - 後から調べる
  - トラブル対応以外にも使われる
    - アクセス解析
    - 問い合わせ対応での使用
    - 労働状況の管理
  - 普段から見とくと、異常に気付ける
- ログの種類
  - パソコンに記録されているログ
  - Webサーバーに記録されているログ
- 何が記録されているか
  - タイムスタンプ
  - ログレベル
  - コンポーネント：どこで起きたか
  - メッセージ：何が起きたか
  - コンテキスト：不可情報
  - 上記のように1つの事が1行で出力
- Windowsのログを確認するには
  - イベントビューアーを使う
    - マウス操作で確認可能
  - コマンドで抽出するのも可能
- macOSのログ確認
  - logコマンドを使う
- ログを取得（記録・管理）するには
  - 優先順位を意識して取得する
    - すべてのログを記録するのは、現実的ではない
    - 記録したとしても、それを分析できるか
  - ログ管理システムを使う
    - 自社の目的に合ったシステムを選ぶ
- ログが無いと、調べようがない
  - ログ監視で、異常を即検知可能
  - ログは、未来の自分（または、チーム）への手紙
- トラブル発生⇒ログの確認⇒現象の再現⇒対策の実施
- 残してはいけないもの
  - パスワード、認証トークン
  - クレジットカード番号
  - マイナンバーなど個人識別情報
  - 機密性の高い社外秘の情報
  - 過度な量のデバッグ情報（本番環境）
  - 意味のない繰り返しログ
- 適切なログレベルで出力する
  - DEBUG：変数の値や処理（関数）の開始・終了：開発環境のみ
  - INFO：正常な動作の記録、アプリの起動や終了：開発・検証環境など
  - WARN：推奨外の動作（非推奨の関数など）：全環境
  - ERROR：全環境
- 複数のログを「つなげて」読む
  - 時系列で並べて読む：時刻情報の動機は必須
- ログを集計する
  - 正常な状態を把握していないと異常には気付けない
    - 時間軸での集計
    - グループでの集計
    - 環境での集計
  - 何が問題を起きて来ると、ログを検索する
  - ログを可視化する
- 問題を発見するパターンの例
  - 突然のエラー
  - 徐々に遅くなる
    - メモリリーク
    - 利用者増
  - 特定ユーザーだけ
    - データ不整合
    - 権限設定ミス
  - 定時に発生
    - 処理の競合
    - タイムゾーンの問題
  - 一部リクエストだけ
  - 再起動で回復
    - メモリ不足
    - 外部環境の不具合
- ログ分析で生成AIが得意な分野
  - 自然言語での要約
  - パターン認識
  - 原因の推論
  - 対話的な調査
- AI：専門家が分析する結果に近い内容が一瞬で出力される
  - 正確性は、検証が必要
- 異常なログの出力を把握する
  - AIに判定させる：例：決定木を使って、AIで自動的に判定する
- サービスの改善に使う
  - パフォーマンスを最適化する
    - 感想：AWSの試験で見るような内容
  - システムが壊れる前に兆候を検出する
    - CPUの温度が高い
    - エラーメッセージが出ているなど
  - ユーザーの行動を分析する
- ログで文化を変える：駄目文化
  - 動いてればOK
  - 障害は、運が悪かった
  - ベテランだけが修正できる
  - 本番環境は触らない
- Q&A ログによって形式異なるので、一元管理してSIEMなどで相関分析したい場合どうすれば良いでしょうか？
  - どのシステムからどういうログが出ているか把握する
  - 前処理をする
