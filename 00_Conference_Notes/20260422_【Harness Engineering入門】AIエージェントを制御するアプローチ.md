# ハーネスエンジニアリングの概要と設計思想	渋谷 優介さん (@sergicalsix)
- ハーネスとい用語の歴史
- ハーネスエンジニアリングのスコープ・キーワード
  - 対象範囲は広く、要素は多岐に渡る
- ハーネスの構成要素とソフトウェア開発の蓄積
  - Principlesの設定
  - Verficationの設定
  - これまでのソフトウェアの開発の上に成り立っている
- ハーネスは、要件定義からテストまで大きな影響範囲を持つ
  - 特に既存のハーネスは、開発・テスト領域に集中している
- ハーネスの設計思想
  - グラフのようにまとめている（レジュメ見よう）
  - 指標例・開発自由度・難易度
- ハーネスエンジニアリングで十分？
  - 不足
    - 開発・テストが早くなっても、デプロイ頻度を上げなければ、ユーザーへの提供価値が加速しない
    - あくまでLLMが働きやすい仕組みづくりであり、人が働きやすい環境を作ることも中長期的な目線で必要である
- ハーネスの事例
  - 要件定義：投機的な開発―並列開発によるdiscovery業務高度化
  - デザインシステム：ユーザー体験原則を含むデザインシステム・事業KPI含むプロダクトPrincipleを用いた確度の高い開発
# 目的ファーストのハーネス設計 - ハーネスを作る前にやるべきこと	Gotaさん(@gota_bara)
- ユーザーハーネスの話
- 現場で良く聞く話
  - 足せても削れず、ハーネスは肥大化し、やがて変更できなくなる
- 判断基準がないまま太ると何が起きるか？
  - 入力トークンの増加
    - 費用
    - レイテンシ毎ターン積みあがる
  - 精度の悪化
- ハーネスを分解する
  - HOWはたくさんあるが、業務目的と状況で最適な選択肢は変わる
    - 業務目的から逆算する必要がある
  - Guide（事前に導く）
  - Sensor（事後に検知）
  - 決定論的・LLM推論的で書く場所が違う
- ハーネスを足す前に考えるべき4ステップ
  - １：業務目的を言語化
    - 何を守るか？
    - 何を守らないか？
    - どこで人が介入するか？
  - ２：構造で守れないか問う
    - 設計・型・依存
    - scaffolding>test>指示
    - 違反が書けない形にできるなら最強
  - ３：指示で補う
    - 残りのCLAUDE.md/skill/review agentは、最小限
    - 指示を足すのは、簡単…判断基準は削るためにある
      - Guideは、どれが効いているか見えないので、削除判断しにくい
  - ４：削れる状態で運用
    - 小さく始める/失敗/駆動/目的で取捨選択
  - 曖昧さや指示に従わなくなって来た時にループして考える
# エージェントの開発環境を内製して気づいた「これもハーネス」	r.kagayaさん(@ry0_kaga)
- 創業半年のシードスタートアップの考え方
- 半年弱は、1人でプロダクト開発を担っていました
- ハーネスエンジニアリングの難しさ
  - 何をすべきか分からない
  - 効いているか分からない
  - いつ消えるか分からない
- ルール・スキル整備は、一部
  - 評価・Evalこそが長期的な改善を導く
  - ターミナルの外へ、個人の活用を越えて
- 一番好きな考え方、「モデルでなければ、ハーネス」
- 「行動を制約し、正しいか確かめ、次をもっと良くする」を1つのパイプラインに
  - Claude Agent + TypeScriptで構築
  - Issueから開発を起動できる＞プルリクまで
  - 再現性と属人製の排除
  - 品質保証の組み込み
  - 計測と改善
- 画面から開発をトリガーするUI Annotator（Chrome拡張）を作る
  - ビジネス側が使用できるように作ってみた
- フィードバックサイクルの自動化を試し中
