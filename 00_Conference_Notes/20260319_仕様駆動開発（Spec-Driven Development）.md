# 12:05~12:25	SDDとうまく付き合うために試行錯誤して見えてきたチームなりの使い方	meimeiさん
- 開発当時の背景
  - 複数リポジトリを横断する開発が多い
- SDD（Spec-Driven Development）
  - Spec
    - What
    - Why
    - HOW
  - Specドキュメントを元に開発する
- Kiro
  - 当時は、Kiro以外で実現できるツールを見つけられない
  - 再現記事を見て、導入
- メリット
  - 漏れている仕様を事前に洗い出し、確認しておける
  - 設計方針が定まっており、各自が迷いなく動きやすい
  - 実装前に決めるべき点を固めて合意済みなので、手戻りが激減
  - AIに精度の高いテストケース観点を出させるため
- 運用する中で見えてきた課題
  - ドキュメントが膨大
    - 最低限の内容を別途説明用にサマリーとして作った
  - 非同期レビュー
    - 改善策
      - モブ作業の定期開催
        - 毎日1時間全員で同期で作業できる時間を確保
        - 基本は、Specドキュメントのレビュー
          - 開発者への説明時間
        - 気軽に相談できる場にもなっている
  - 複数システム連携
    - コンテキストを渡しづらい
    - 各リポジトリで独立してSpecを作ると、連携品質が落ちる
    - 改善策
      - サブモジュールとして、各リポジトリのSpecをまとめた共通Specを配置
        - 各システムのリポジトリには固有のSpecを置く
        - 各リポジトリには、統合リポジトリへのシンボリックリンクを貼っている
  - どのツールを使うか
    - 現在は、多様なものがある
    - どれを使ってもプロセスは、変わらない
    - 当社のように自作も手
  - Specはいつまでメンテナンスするか？
    - 機能開発中における合意形成のためのドキュメント
    - 開発後は、コードが正
  - planモードとの使い分け
    - 複雑性の高い開発では、Planでは粒度が粗い・カオスになりがち
  - 結論
    - 誰にでも、どんな時でも共通するSDDの正解は無い
- Q&A 既存？新規
  - 既存で開発していた所に導入
- Q&A forkした？
  - 完全に自作（当時は無かったので）。似た物には、なっている
# 12:25~13:05	生成AI時代のドキュメント基盤	中村充志さん, 山浦哲朗さん
- ドキュメント体系：どう構成し、どのように役割分担するか
  - いつ
- ドキュメント基盤：どのように配布するか
  - どう作成し、配布するか
- ドキュメント基盤のきっかけ
  - 7年前に6時間かけたWordが崩壊してキレた
    - markdownベースに移行を上司に説得した
      - 結果的に、次世代の生成AIに合う物だった
- 実績
  - 800ページ超
  - 大手で実践
- ドキュメントの要否は、プロジェクト次第
  - ドキュメントとコードの役割は別
    - ドキュメント＝コードの一致は、「契約」の一致
    - 「契約」が必要かどうか
    - 立場が違う議論は、検討違いになりやすい（役割が違う）
- ドキュメントの維持
  - お金とプロセスがあれば、維持可能
  - AI使えば、手をかけない事も可能
- ドキュメント基盤要件
  - 「単一PDF出力」入っている
