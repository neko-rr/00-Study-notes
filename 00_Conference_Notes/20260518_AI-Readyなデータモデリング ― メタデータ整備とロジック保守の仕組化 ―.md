# 「LookerとADKで作る社内AIエージェント」	株式会社タイミー 中村 優 氏
- AI時代のデータ抽出アプローチ

    - PdMを中心に分析用のAgent Skillsを運用し始めている
  - セマンティックレイヤー
    - BigQueryをデータウェアハウス、dbt～
- どちらかが正しいというよりは、適材適所で使い分けるべき
- タイミーでは、指標の一貫性が重視されてきた
  - コンサンプションモデルで、クライアントに使い続けてもらうことが重要
  - カスタマーサクセス担当者が、利用状況を確認しながらクライアントと伴走
    - 全員がSQLを読み書きするのは、ハードルが高い
  - ビジネスを適切に表現する指標定義が複雑になるケースが多々ある
    - 例：切り口が複数
- Lookerを通したデータ活用が当たり前に
  - BigQueryをデータウェアハウス
  - dbtでディメンショナルモデリング
  - Lookerで可視化する構成を整備してきた
- 課題
  - Lookerでデータ抽出するまでのステップが多い
  - 事前定義されていない切り口や指標を出力できない
  - 営業組織で利用できるAIサービスが限定的
  - AIを取り巻く進化は、目まぐるしく、ベストプラクティスが確立しにくい
- 指標出力エージェント：メトリっち
  - ゆるキャラ（笑）
  - Slackで応答
  - GoogleのADK
    - メトリっち以外のAIエージェントもデプロイできるモノレポ構成
    - PRごとにプレビュー環境を構築
      - PRごとに全部試して、性能を比べるため
- adk evalを使った評価
  - ツール呼び出し
    - ワイドテーブルを正しく選択できているか
    - フィールドを正しく選択できているか（表示、フィルター、ソート等）
  - LLM-as-Judge
- フィードバック機能
  - ユーザーが会話を評価して、質問に答えられなかったケースを把握して改善に繋げられる
    - 結構、学びが大きい
      - 実は、○○のデータがよく検索される等
  - AIの挙動というよりは、そもそも今のセマンティックレイヤーで調べれない内容だったりする
- 成果
  - 社員誰でも使用できる
  - ベストプラクティスを追求しやすい
- 個人的には、ドメイン知識は、skillに寄せて行って、全部再現したい
# 「分析エージェント精度向上におけるデータアナリストの役割」	合同会社DMM .com大浦 将弥 氏
- DMMにおけるデータ組織
  - 基盤開発
  - アナリスト
    - 効果検証
    - 分析エージェント
  - サイエンティスト
    - 機械学習アルゴリズム
      - 検索
      - レコメンド
- Slackから分析問合せ
  - 簡単な統計処理
  - グラフ描画
- 更に発展した分析エージェントを現在作成中
- 精度向上例
  - AIが勝手に判断して勘違いで進んでいく
    - プロンプトの聞き返し機能
      - プロンプトの質と出力結果は、比例する
      - アナリストが、事業部と過去にやり取りして得た知見を知識として実装して、判断基準を作っている
  - DMMは、60個以上のサービスがあるので、同じ単語で別の意味がある
    - Knowledge Catalog（Googleクラウド）に一元管理
      - 用語の定義が変わる時は、データアナリストの承認必須にして、管理
      - 現場担当者への浸透もさせている
  - ゴールデンクエリ
    - メタデータ＋典型クエリの整備
    - 改善サイクルも組んでいる
  - 真に知りたい事は、利用者のプロンプトに出てこない
    - 因果推論
      - 非常に難しいタスク
      - 過去の分析知見の整備
        - データアナリストの過去レポート
      - 人間が対応しても難しいタスクは、無理だったなら、ばっさりと切るのが安全
- 各社分析エージェントを作成していると思うが、データエンジニアだけでなく、アナリストと一緒に行う方が良い
  - 人間が過去に間違えた事は、AIもやる
