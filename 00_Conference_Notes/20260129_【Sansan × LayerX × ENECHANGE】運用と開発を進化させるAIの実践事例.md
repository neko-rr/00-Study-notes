# ②システムのアラート調査をサポートするAI Agentの紹介(仮)
- 「バクラク」
  - アラートタイムにおける課題
  - AWS使用
    - サービスが100以上ある状況（ECSサービス）
    - 監視にまつわる前提
      - 大半のシステムにおいてはDatalog
    - プロダクト間連携も多数行われていて複雑
    - アラートの調査とコードの修正も行う
    - プロダクトの理解が無いとアラート調査ができない
      - 新入社員へのアラート調査の指導コストが高い
- アラート対応の改善検討
  - アラート調査は一定定型化する
  - 現在、内製化予定
    - 他にAWS DevOps Agent：検討したが、ＡＷＳログイン限定＆スラックからの調査依頼ができない（報告は可能）
    - Datadog Bits AI SRE：検討したが、全ての調査がDatadog外の。。。
    - 内製化きっかけ：社内のAI Agent開発を行うハッカソンを開催した
      - CloudWotch利用のエージェントを作成した
      - システム調査エージェント：PoC中
      - 使用技術
        - Python
        - Strands Agent
        - Datadog MCP Server
        - CloudWhatch MCP 
        - 他も
      - ユーザーはSlackで依頼して、探って、Slackに報告で完了
        - 統括エージェントとMCP問合せエージェントを分割
          - 責任の明確化し、追加エージェントも増やす時も対応可能
          - 入力トークンの管理
          - 調査レポートにおけるタイムウィンドウの統一
            - 明確な指示が無い場合は、日本の現在時刻で調査してください
- 今後は、調査毎の特有のコンテキストを与えていないので非効率な調査している事あるのが要改善
# ③Claude Code Actionsを用いてアラートの初期分析の効率を上げたい！(仮)
- 従来のアラート対応フロー
  - Sentry⇒Slack⇒Cloud codeで用意していたスラッシュコマンドで分析⇒
- 課題
  - 共有の壁
    - 調査結果は実行者のClaude Code上にしか表示されない
    - 他のメンバーに共有するかは本人次第
  - 待ち時間
    - 中途半端に10数秒
- 解決：Claude Code GitHub Actions
  - 手動で行っていたコマンドを自動化
  - 必ずSlackに報告まで自動化
- 工夫
  - トークンを沢山使用しようとするので、制限する工夫を入れている
  - 権限を明示的に指定する
- アーキテクト
- 今後の展望
  - 過去の類似エラー調査：以前できていたが、今、あえて制限中
  - Slackに解析結果まで表示：今、GitHub Actionsのサマリを確認する必要がある
  - BugSangなど社内使用のツールに接続させたい
- Q&A：同じアラートや同時アラートの制限しているのか？
  - Sentry側で抑制可能
  - 後、長く続いているから、安定していて、そんなにアラート出ない
- Q＆A効果
  - 初期の当たりを付けるのが楽になった
