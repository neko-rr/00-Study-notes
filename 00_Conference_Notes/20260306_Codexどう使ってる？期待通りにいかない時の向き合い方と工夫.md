# Codexに役割を持たせる ─ 他のAIエージェントと組み合わせる実務Tips(仮)	o8nさん (@o8n_project)
- 失敗編
  - タスク設計とレビューで失敗した
  - Cloud codeと同じ感じで、丸投げした
    - 確認なしで突き進み、方向修正できなかった
  - 実装もレビューも同じAIに任せた
    - バイアス入って、見逃す
  - AGENT.mdに全部書きすぎた
    - READMEのように開発情報を全部記載
    - 指定と違うコマンドを使う、規約を守らない
    - コンテキストが埋もれて、重要な指示が届かない
- 「役割」と「ワークフロー」を整えた
  - タスクを小さく切り、AIごとに役割を分ける
  - Skills/Commandsで手順を再現可能に落とし込む
- 改善編
  - Claude Codeをドライバーにする
    - Claude Codeが計画・実装・修正・PR作成
    - Codex
      - read-onlyレビュー
        - 計画も最大3回レビュー
        - コードもレビュー
    - GitHub Copilot
      - PR自動レビュー
    - 複数視点が入れられるのが良い
  - Skillsでワークフローを組む
    - Claude CodeのSkills/Commandsに各工程を定義
- Codex開発者へのインタビューから見える現場の実践
  - Codex自身がCodexのコードの90%以上を生成
  - エンジニア1人が4～8の並列エージェントを運用
  - AIレビュー⇒重要コードは、人間レビュー必須の階層的レビュー
- Harness Engineering
  - OpenAIがCodexを用いたagent-first開発の実践知
  - コンテキスト設計
  - 制約のコード化
    - カスタムlint・構造テスト
- Codexの進化が速い
- なぜCodex CLIに「レビュー役」を任せるのか
  - 実装とレビューで異なるモデルにしたい
  - OSSの定番パターンに強い傾向がある（所感）
    - 社内コードへの異なる視点として有効
