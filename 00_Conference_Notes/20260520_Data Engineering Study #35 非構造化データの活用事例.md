# 犬塚 眞太郎 氏「運転動画を検索可能にする 〜Cosmos-Embed1とDatabricksで〜」
- 発表資料：Twitterに挙げ済み
- 来歴
  - 2019年新卒クックパッド
    - リサーチエンジニア⇒検索エンジニア
  - 2022年サイバーエージェント
    - 機械学習エンジニア
    - ABEMAでのCM広告最適化
  - 2024年チューリング
    - MLOpsチームで自動運転のための基盤づくり
- E2E自動運転を内製開発する日本のスタートアップ
- 運転環境は「ロングテール」なデータ分布
  - 交通環境には、頻度が少ないが、多様で困難な状況が存在する
- 欲しいシーンが見つからない問題
  - 数万時間規模の運転動画が蓄積
  - 3チームから同時多発的な要望
    - シナリオテスト拡充
    - 学習データ品質管理
    - モデル挙動分析
  - 共通項：映像の中身をキーにシーンを探したい
- 作ったもの：シーンのセマンティック検索
- Cosmos-Embed1
  - NVIDIAが2025年に公開
  - 10億パラメータのvideo-text joint embeddingモデル
  - 映像とテキストを同一ベクトル空間に射影
  - 学習データ：約800万動画
    - 自動運転、Agibot/BridgeV2など物理世界データセットを含む
  - 物理世界（自動運転・ロボット）のデータに特化
  - NVIDIA　Cosmosプラットフォームの一部として商用利用可
  - 他のモデルと比べて、時系列に強い
- 大きいモデルなら勝てるか？
  - Qwen3-VL-Embedding-8Bも検証
    - 走行動画で、「intersection（交差点）」すら正しく識別できない
    - 推論時間・メモリは、大幅増
    - 汎用モデルとしては優秀でも、走行シーンに特化したembeddingが出ない
    - 汎用モデルのスケールアップが必ずしもドメインタスクの向上には、繋がらない
- Databricks NotebookでのPoC
  - 1走行（約1時間）分のシーンから試した
  - 既存Delta Tableのメタデータ（車両ID・日時）をそのまま活用
  - embeddingをparquetに変換⇒Vector Search Index作成までNotebook上の数十行で完結
    - 社内展開もかなり楽
  - インフラ構築ゼロで本格検索まで到達
- アーキテクチャ
  - AWS
- 設計ポイント
  - 計算コストの非対称性
    - Vedeo EmbeddingとText Embeddingの計算コストは大きく違う
      - 同じ扱いをしない
        - Vedeo:事前計算してインデックスへ
        - Text：リアルタイムで都度計算
      - モデル推論が必要かで、経路を分岐
        - テキスト検索：Lambda経由
        - 類似シーン検索：Databricks直接
- 当初は「映像⇒embedding」を1パイプラインで構想
  - デコードが思ったより遅い
    - 実際に計測してみると動画デコードの方がGPU推論より重い局面があった
    - GPUを確保して、デコードに使うのは贅沢
  - ボトルネック分離
    - Fargate CPUでデコード
      - H100で推論専念
- フレーム画像抽出パイプライン
  - embedding生成とは、別ジョブとして分離
  - AWS Batch（Fargate CPU）でフレーム抽出
    - 8フレーム切り出し
    - WebP圧縮
    - tar.gzでS3へ
  - tar.gzでS3GETリクエストを1/8に削減
  - サンプリングレートは、モデルごとの設定ファイルで差し替え可
- 日次Embedding計算
  - オンプレH100 + Slurm
  - AWS P5オンデマンドよりオンプレが圧倒的に安い
  - 毎日2:00にSlurmジョブ起動⇒学習と競合せず
  - 未処理シーン検出
    - S3での集合差分だけ
    - H100のGPUを使い切るための工夫もしている
- Databricksにある程度お任せで、テキストベクトル計算が可能
- なぜテキスト検索にLambda
  - 時間と利用頻度
  - Lambdaを使いまわすための工夫もしている
- 検索UI
  -　JADD Studioへの組み込み
  - 社内ツールへの統合
  - 類似シーン検索もある
- 後追い改善
  - 車両・日次フィルタ
    - Vector Searchのフィルタ機能を活用
- コスト
  - オフライン日次
    - オンプレで減価償却分のみ
  - Vector Search Endpoint
    - 数十ドル/月
  - Text検索Lambda
    - 利用頻度に応じた変動費のみ
- まとめ
  - 結構、データ基盤側に全乗りで作れるようになってきた
## Q&A
- データが増えて来ると、コスト増大しますか？
  - ここ2～3年くらいは、月数十ドルで行けると思う
  - でも、会社特性でデータ量が多いので、そこらのS3に比べたら可愛いもの
- どれくらいでPoCしました
  - 1週間で作って、1週間試して配布
  - ただ、Databricksの使い方で罠を踏みまくりました（笑）
- 精度
  - そもそも海外の動画で学習しているから、右側通行・左側通行がある
  - 今、データを集めまくっている所で、精度と改善は、これから
- 実験
  - 実は、車種によって挙動が違う：ブレーキの利きとか
    - だから、特定の物だけ調べたくなる
    - ハードウェアって、大変だなと楽しめる人は、是非
- Databricksを使用している理由
  - スタートアップなので、色んなツールに移動したりするから、ロックインがされにくい
  - 移動しやすい

# 名取 亮 氏「SnowflakeとAIで作る、SaaSエンジニアのエスカレ対応効率化エージェント」
- バクラク申請・経費精算の開発エンジニア
- お客様のお問合せ対応
  - カスタマーサポートからエンジニア対応必要なら来る
    - エスカレーション対応専任が居る訳ではないので、日々の突発対応が辛く、「お問合せ対応エージェント」開発ギルドの発足
      - LayerXでは、チームを横断して特定のテーマに関心を持つメンバーが自発的に集まるエンジニアコミュニティ「ギルド」がある
- エスカレーションは、Slack上で行われる
  - Slackボットにしよう
  - 猫アイコンのRamseyお問合せサポート
  - リンクは、Streamlit in Snowflake（SiS）アプリへのリンク
    - エージェントの調査結果を表示するSiSアプリ
      - 以下の結果を参考にエンジニアが追加調査を行う
        - 問題の分析内容
        - 根拠が不十分な点・自信度
        - 推奨される次のステップ
        - 過去の類似問い合わせへのリンク等
- 人のやる事をそのままAIに置き換えた
  - 分析用のSnowflake
    - 活用課題
      - DWH上のデータベーススキーマは、システムの都合で構築されている
      - お問合せは、自然言語
      - 自然言語と実データスキーマとのギャップ
    - 解決
      - SnowflakeのCortex Agents
      - 自然言語を投げると、Snowflakeが自分でSQLを生成・実行して答えを返すエージェント機能
      - 作成したエージェントは、チャットUIの他に、APIからも利用できる
      - データがトレーニングに利用されることもない
      - Cortex Agents単体としても利用できる
- Cortex Agents
  - LLMが2種類のアプローチでデータを参照
  - Cortex Analystのセットアップ
    - GUIあるいは、SQLを作成
    - 調査と分析用途では、使い方が少し異なる
  - Semantic Viewの作成自動化
    - バクラク経費だけでも100以上テーブルあるので、手動作成は無理だった
- 過去事例の活用
  - Zendeskチケットの検索
  - RAGもCortex Agents（w/Cortex Search）
    - 公式推奨
- お問合せエージェントの改善ループを回す
  - 外部へのデータ送信制御：Snowflakehは、管理者しか変更できない
  - エージェントの出力のアクセス制御
    - Snowflakeの権限管理に乗っかる
    - データ利用の透明性・監査
      - toolで利用したSQLを残す
      - テナント指定が曖昧なセッションに警告
- 今後の課題
  - 生産性のメトリクス（調査工数）には、まだ改善が見られない
  - 最近、Cortex Agent Evaluationsの検討

# 三浦 諒一 氏「AI-Readyなデータ基盤を構築するメダリオンアーキテクチャ」
- AIエージェントの市民活用
  - Difyを社内展開
- 活用が進むにつれて、エージェントに与える「データ」の品質が良くないとエージェントの回答精度が悪くなるという課題
  - マニュアル等の非構造化データの加工を自動化したい
- メダリオンアーキテクチャ
  - データレイクハウスにおいて、データを3段階に分けて、行き来できる手法
    - 今回、データを触る人のレベル感がこれに丁度良かった
      - ITチーム：Bronze
      - AIエンジニアリングチーム：Silver
      - データエンジニアリングチーム：Gold
- 業務基盤：Microsoft365
- データ分析基盤：Snowflake
  - データアナリストはSnowflakeがメイン
  - AIエンジニアはDatabricksgがメイン
- SharePointへの接続
  - 同じ会社のMicrosoftからの方が相性が良かった
- 大きく、PDFとPDF以外に分ける
  - PDFは、PyMuPDFで各ページをマークダウンで抽出
  - 当初：PPTXなどのオフィス系のファイルは、当初MarkitDownというPythonライブラリを使っていたが、文字以外が駄目だった
    - じゃ、全部PDFにしよう
      - Fabricの仕様上インストールできない事が判明
        - 詰んだと思ったが、MicrosoftのGraphAPIを使うとPDFに変換された状態で取得可能と知った
- 工夫した所
  - その要素がどのページかを反映させた
  - Difyでは、画像URLがあると回答に画像を表示できるので、回答内容の根拠として提供可能
    - だから、画像をシェアポイントに戻している
- Snowflake Managed MCP Serverを使用して、提供する
- まとめ
  - やはり、権限が難しい
  - 内容、テックブログに載せています
