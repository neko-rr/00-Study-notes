# 株式会社Legalscape 小林 賢	インフラハーネスエンジニアリング
- AIに開発だけでなくインフラ管理・運用も全部やらせたい
- terraformは、ハーネスの土台として望ましい性質が多い
  - テキストベース
  - 宣言的：人間がレビューしやすい
  - plan⇒applyの2段構えがハーネス的
  - 枯れた技術
    - AIが良いコードを書きやすい
- 権限管理が難しい
  - 落とし穴
    - workflow自体を書き換えると、terraform applyを実行できる
  - Atlantis
    - PR上のコメントでterraformできる
    - 外部のサーバーで安全
# 株式会社TOKIUM 坂上 晴信	AIを賢くしたいなら、まずは人間の改善ループから
- チケットからPull Request作成まで自動化
  - jira-to-prスキル
    - 3チームで3名ずつ開発していて、かなり速度UPしている
- コードレビューの自動化
  - 2系統に分離して、目的別に回している
    - セルフレビュー
      - 自分のPull Requestに対して
        - コード的な修正
      - 他メンバーのPull Requestに対して
        - 意図を収集
        - マージ前のガード＋意図の整理・構造化
- ノウハウ
  - 失敗しても良いところから入れる
  - AIの思考過程を記録し、観察する
    - スキル実行の過程
    - 集計・基盤
      - 何処に置くかも意識する
  - 課題点をこまめに修正する
    - 週1回、ハーネスも含めたプロセス改善が必ず実行される状態になっている
    - 細かな改善をこまめに取り入れる
# 株式会社estie　武田 純真	AI が大量に生成するコードを人間がレビューしやすくするためにやってきたこと
- レビュー：AIのコード内部品質の低さに由来する認知不可が大きい
- コーディング規約を細かく整備
  - CLAUDE.mdに細かく整備
    - ～1000行
    - 内部品質は、一定上がったが、規約に従っていないものがある
      - 恐らく、コンテキスト多すぎ＆最初しか読み込まないから
- 規約のうち静的解析で弾けるものは、linterに移管
  - 残りは、AIレビュー
# 匠技研工業株式会社　小森 詠史	ハーネスエンジニアリング - 概論を踏まえて何からやるか
- AIエージェントが自律的に動くために、周辺環境を設計・整備する事
- Claude Codeにプロジェクトの課題と解決策の草案を作らせる
  - エンジニアたちで、検討
- 評価軸を決める
  - 即効性・決定論性・既存資産の活用・人手削減・独立性
  - 優先順位を決める
- pre-pushの実行から修正までを決定論的に行う仕組みを作る
  - エージェントの終了時に未コミット変更がある時は、型チェック等の確認をして、修正してからタスクを終了させる
- 失敗をルールへ昇格する仕組みを作る
  - 直近1ヶ月分のクローズ済みまたはマージ済みPRから、レビュー指摘を収集させる
    - 2回以上出現する類似から、ルールを考える
- 受け入れ基準をテスト化して、テスト駆動開発する仕組みを作る
  - skills
- どういう風にやるか
  - リソースを確保する
    - 特定のエンジニアを一定期間確保して進めるのがオススメ
  - 開発生産性への投資として、エンジニアでなくても分かる言葉で説明する
- 成果
  - レビュー漏れの修正が減った
# 株式会社mento 門脇 恒平 (kadoppe)	AIプロダクトのゼロイチ開発を支えたハーネスエンジニアリングの変遷
- プロダクトや組織のベースによって、必要なハーネスが変わりますよー
- プロトタイプ開発
  - 原則、使い捨て・メンテナンスは考えない
  - やりたいこと
    - 頭の中のアイデアやイメージを最短で具現化
  - ハーネス
    - 基本的なLintとか、フレームワーク知識とか
- 本実装
  - 長期的な保守性・可読性も考慮した開発へ
  - チームメンバー間で共通地を作る
  - AIにも理解できるようにする
  - ハーネス
    - ドキュメント整備
- リリースに向けた品質改善
  - AIプロダクトの品質は、壊れやすい
  - コード変更の良し悪しの評価が難しい
  - ハーネス
    - 考えうるユーザーペルソナ像20名ほど生成
      - 各ペルソナがAIコーチングを自動利用して内容を評価するLLM as a Judge
