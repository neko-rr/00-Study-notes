# 個人開発を丸投げしたら見えた Claude Code に任せていいこと、ダメなこと	スナガクさん (@suna_gaku)
- 機能実装からPR作成まで全部任せられる
  - Planの精度がカギ
  - Human In the Loopを随時入れる
- 要望ベースで実装できる
  - 最近、「こんな機能が欲しい」から始められる
    - 事前に画面を作らせて認識を合わせる
- Agent Teamsで開発以外のタスクも任せられる
  - AskUserQuestionで暴走を防ぐ
- 任せられない事
  - 古いバージョンのライブラリを使わせない
    - モバイルアプリは特に要注意
      - 必要に応じて、Context7を使用
      - Auditエラーの検出をCIに入れておく
  - 動作確認の永遠ループを防ぐ
    - AIがエラーを認識せず同じ修正を繰り返す
      - 利用制限まで使い倒す。。。
        - 試行回数の上限を設定する
          - 「二回試してダメだったら、バグとして報告して」
          - ライブラリの不具合を疑う
            - どれだけ修正しても直らないから、ここは人が検知する必要あり
  - 禁止事項をAIに伝えておく
    - 「型エラー直して」で全部anyにされる
      - Lintでno-explicit-anyを有効に+「anyにさせないで」と伝える
    - 指示だけでなく、背景とやり方まで伝える
  - Skill読み込み途中でCompactionを走らせない
    - 途中まで読み込んだ内容がリセットされる
      - 予期せぬ挙動をする可能性がある
        - StatusLineでContext消費量を可視化
          - タイミングを検知して、事前に手動Compactする
  - 保守性の高いテーブル設計
    - 実例：タスクの状態カラムを3つ作られた
      - 同じ役割のカラムが3つ存在し、データ不整合のバグが永遠に治らない
      - E2Eテスト全落ち、修正に2日
  - DBマイグレーションは、必ず人が検証する
    - AIは実行エラーの有無しか検証できない
      - データ構成が正しいか、ドメイン的に問題ないかは確認不可能
    - 既存データとの整合性に特に注意
      - FK参照が壊れてデータ消失
        - AIは、作成順序を考慮しない
    - 対策：ローカルで必ず検証する
  - マイグレーションファイル作成コマンドを作る
    - 直接やられると履歴がの
# Claude Code利用ログ基盤の構築	giginetさん(@giginet)
- 仕事はLINEの開発。スポンサー枠
- ログの取り方
  - １：Open Telemetryを使ったモニタリング
    - 使用量しかわからない
  - ２：チームダッシュボード
    - 変更行数しか分からない、権限管理の問題
    - 会社で契約している場合使用可能
  - ３：Claude Code Hooksを使ったログ送信
- ３：Claude Code Hooksを使ったログ送信
  - Hooksを使ってイベントがあった時にログを送信
  - Hook Lifecycle
    - 公式ドキュメントにhookがまとまっている
    - 新しいログが増える度にダッシュボード増やしている
  - Payloadの設計
    - その他メタデータの追加
      - 現在のリポジトリ、hostname、現在のブランチ等
    - 実際のプロンプト等も取れるが、それは取っていない
    - プラグインのデフォルト有効化
      - リポジトリで強制的に有効化可能
    - トラッキングしているメトリクス
      - MCP/Skill/Subagentの利用状況（呼び出し回数、ユニークユーザー数）
        - 何のSkillが使用されているかまで分かる
      - ユーザーごとのプロンプト発行数
      - Edit/Writeの変更行数
      - Instructionのロード回数
      - Planモードの利用状況（ユニークユーザー）
- 生産性の計測
  - 利用量＝生産性ではない
  - 他の生産性指標と結び付けて、考える

# ハッカーよりもClaude Code 〜コンテスト日本1位から見るセキュリティをAIで担保する時代〜	Satokiさん(@satoki00)
- Clude codeよりも脆弱性を見つけられますか？
  - 国家レベルハッカーより下は、Clude codeが勝つ
  - 診断依頼してくる人は、まだClude code使用していない人が多かった
    - だから使いましょう！
- dangerously-skip-permissionsで全権委任
  - 権限認可を挟むと、許可されたツールに固執して性能が低下する感覚
  - 試行速度・回数が命の脆弱性診断では、権限の確認時間がボトルネック
- 「朝8:00まで絶対に診断を続けて」⇒寝る
  - 指示を仰いで、途中停止するのを防止
- 既に手動検査は、意味が無い
- 脆弱性診断⇒脆弱性トリアージの時代になっている
