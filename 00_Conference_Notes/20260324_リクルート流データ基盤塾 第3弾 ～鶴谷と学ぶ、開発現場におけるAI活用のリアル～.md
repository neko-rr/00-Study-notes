# AIエージェントフレームワークの比較検証
- AIエージェントの説明
- AIエージェントフレームワーク
  - LangGraph
    - ワークフローをグラフとして定義し、独自のAgentループを構築する事が可能
    - カスタマイズ性が高い反面、学習コストが高い
    - グラフ管理もあり、運用コストが高い
  - STRANDS AGENTS
    - AWSが開発・提供
    - AWSとの高い親和性
      - AWS以外でも使用できるが例のドキュメントが少ない
    - コード短いし
  - Agent Development Kit
    - Google作成
    - Google Cloudと親和性が高い
    - 少しカスタマイズが必要
  - OpenAI Agents SDK
    - 非常に簡単
    - 複雑なワークフロー構築には不向き
- 事例紹介
  - SUUMO：不動産マッチングプラットフォーム
    - AIエージェントを用いた業務支援ツール
      - データの要約・業務の進め方についてのアドバイスを生成するツール
        - デリバリー最優先のため、GoogleのADKを深く考えずに使用
- 課題
  - 不要なマルチクラウド
  - フレームワークの調査不足
- 結論
  - STRANDS AGENTSを採用
    - AWSを元から使用していて、他の方が良い利点が無かったので
  - 他の2つのプロジェクトでは、複雑性が求められたので、LangGraphを使用した
# AIエージェントによるプロダクト運用の自動化 - 社内横断プロダクト Crois におけるAIOpsの実践
- 横断データ基盤Croisを新卒2022年入社からずっとやっている
- Crois
  - 内製プロダクト
    - ワークフローエンジン・ジョブスケジューラ機能
    - 稼働規模
      - 20,000/日
    - 横断的なデータ活用施策に不可欠なインフラ基盤
- 運用業務としてのインフラ調査
  - Slack上での利用者から様々な問合せ対応やアラート対応をする
  - クラウドインフラやソースコードを直に見に行かないと分からない、難易度が高い
- 2022年頃から会社統合や別プロダクトとの統合機能により利用量増大
- 従来の運用
  - プロダクト運用効率化の文脈では、手順書の整備・スクリプトによる自動化が代表的
- 今
  - 仕様書駆動開発から「手順書駆動運用」へ
    - 既存の手順書が高品質なプロンプトとして使用可能
    - Slackを使用してリクエストすると、エージェントが対応してくれる
      - 代表例
        - アラート対応
        - ふわっとした事、インフラ調査
        - 問い合わせがふわっとしている時に、そのまま周辺を調べさせてリソースレベルで対応させる
          - 結果的にエラー特定される
    - 成果
      - 調査タスクの負担を軽減
        - 前までは、人間があちこち見ていた（見て、戻るを繰り返す）
      - クロサイの出力を評価
        - 7割は、人間がやるよりも同程度以上
      - Human in the loopによる継続的改善
        - 運用手順書を読み込みしているので、手順書が悪いと判断されるとプルリクが出される
          - スクリプト変えるよりも簡単。面倒な手順書更新が、自動化に直結してモチベアップ
- 結構、簡単に出来た
  - 元から、高品質なテキストがあったから
  - 自動化のためにドキュメントを充実させよう
    - 既に整備済み
  - 自動化のためにシステム監視を充実させよう
    - 既に整備済み
