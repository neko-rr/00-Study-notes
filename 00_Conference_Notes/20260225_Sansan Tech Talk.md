# 成長するプロダクトの複雑性　加畑 博也
- sansanの開発者
  - 2014年入社
- 売上高も契約件数もまだ伸びている2024年～現在
- 企業規模も様々で1名企業から大企業まで
- 成長と複雑性
  - 成長するプログラムは放っておくと複雑になっていく
  - 複雑性対処した事例紹介
- 観測する：特定の時間帯・データベースのパフォーマンスが著しく劣化していた
  - データベースサーバーのCPU/GPU/メモリや、PostgresSQLの共有バッファの空き容量
  - 規模感：バッチ処理1日1億処理
  - 対応：仮説を立てて観測する
    - PostgreSQL内部には軽量バッファが存在する
- 予防する：会議の音声から商談記録を自動で作りたい
  - 機能を作ることはできるが、今後もメンテナンスが必要になる
  - 必要十分を徹底的に考える
    - 提供したい価値を分析し、スコープマネジメントを徹底する
  - 企画・要件定義からアプリケーションをエンジニアが考える
  - 必要最小限MVPで早期にリリース
- 整理する：アプリケーションコードにおいて、DDDをベースとしたアーキテクチャ設計があった
  - モデルとしての登場人物を最小限にし、用途を明確にした
  - 中長期的な開発のコストを織り込み、適切な抽象化で整理しなおす
- ちゃんとコアな課題に向き合って、考え続ける文化
- sansanは、エンジニア25名で発表者が関西はこの人1名のみ。しかも、グループマネージャ。
# AI 時代に考える、Sansan の新規プロダクトのアーキテクチャ　横山 拓史
- Data Intelligence Engineering
  - sansanもBill Oneも開発関わった事ある
- Sansan Data Intelligence
  - 顧客管理システムは、手動で入力するかデータが綺麗では無かったり、入れてなかったりする
    - その手動を肩代わりして綺麗に入力するシステム
  - 当時、わくわくプロジェクトととして始まった
    - かなりAIを意識して作った
- １：コンテキストを近くに置く
  - AIが参照するコンテキスト情報は、AIの実行環境と近い場所に置いた方が良い
    - ドキュメントとコードベースの整合性をとりやすい
    - ドキュメント自体を書く時もAIが使える
    - GitHub Actionsとの親和性もいい
      - ドキュメントが古くなっていないかのチェックを月1で実行⇒自動Issue化
        - 160個プルリク出来た
- 2:全てをAIに任せない
  - 構造的なコードの出力をさせるためには大量のコンテキストが必要
  - DDDをベースとした3層の構造化でコンテキストを整理し、スコープを明確化した
    - BC
      - DU　：デプロイ単位（発表者曰く、挟んだのは失敗だったと考えている）
        - Aggregate
  - ファイル構造
    - 集約で分離している
    - コマンドを叩くと、まとめて増える
  - ディレクトリとしてコンテキストを区切る事で、AIが把握する必要のあるコンテキストのスコープが明確に
  - ボイラープレート側にテストの型も持たせているので自然とテストカバレッジも上がっていく
  - ボイラープレート側にAIへのTODOコメントを記述する
- ３：短いFeedback Loopを構築する
  - AIで大量コードが作れるようになったので、AIが自己チェックできる仕組みを以下導入
    - Lint
    - Service Level Test
      - HTTPリクエスト経由でテスト
      - さらにシフトレスト
  - どれだけシフトレフトできるかが生産性に直結する
  - レビューのようなAIによるチェックをいれるのであれば、別セッション上で評価させた方がより精度が高い
    - コード生成させたのと別セッションで確認する方が良い
- Claude Code使用している
  - Spec Driven Development
# 守るだけじゃない、攻めるITへ：EX（Employee Experience）を"シンプルにする"組織とは　武内 孝憲

  - 現在、コーポレートシステム部を無くす予定
  - 自動化していって、最終的に部を消す
- 部の体制：パートも入れて、約70名
  - 主に社内のアプリ開発
  - ミッション：EXをシンプルにする
    - 社内のあれこれを自動化して、面倒な申請を無くす
    - SimpleとEasyは、似て非なる物
  - 社外に誇れるIT環境を作る
    - 良い物は活用し、無い物は作る
      - クラウド：AWS/GCP/Cloudflare
      - OS：Mac、Windows、Linux、iOS、android
      - 利用言語：Python、Go、…

  - 勤怠申請アプリ:Slack経由でKintoneでデータを登録される
  - 解決方法：この部は、可能な限りサーバーレスでやるので、Lamdaを使用
  - IaC化できる事は、できるだけしている
  - 基本、エンジニアだらけなので、SlackやAPI叩いたりで解決
  - 余談：メンバー1名でアプリ・基盤を構築したことが評価され、昨年末の社内MVP受賞：PoC Sansan MVPサーバー構築
    - 300万管理する名刺があるお客さんの悩み解決
    - 並列でAIエージェントに指示だして、CodeRabbitでコードレビューさせて作成

    - 本質を突き詰める
      - ECRS（イクルス）の4原則
      - 極論、作らずに課題解決できるのが最強・最速である
    - 最強・最速を目指し続ける
      - 言われる前にやるを目指す
      - 攻め
