# 「もしもデータ基盤開発で『強くてニューゲーム』ができたなら今の僕はどんなデータ基盤を作っただろう」	イオンスマートテクノロジー株式会社 櫻井 良輔
- 途中参加
- 基盤体制作る黎明期に求める人は、地頭の良さと向上心
- 昔の自分へ
  - とにかく汎用的に作れ
    - チャッキアップに時間かかる
    - 柔軟性ない
  - 格納するデータは、1丁目1番地から
- データ基盤に収容を諦める理由はいろいろ
  - そもそもデータを収集していない
  - データソースシステム管理者と調整できない
  - 管理システムとNW経路がない
    - データの数をKPIにすると、「球場のトイレの利用率」等の意味のない持ってきやすいデータを入れる事に時間をかけちゃう
- データ基盤作りに必要
  - 本質をとらえる
    - 何か別の目的があって、方法としてデータ基盤作りが始まる
  - 高い視座、広い視野
    - 課題を解決する手段は、一つではない
  - 汗をかき仲間を増やす
# 「モダンデータスタックの理想と現実の間で～1.3億人Vポイントデータ基盤の現在地とこれから～」	CCCMKホールディングス株式会社 松井 太郎 氏
- 営業⇒システム運用⇒開発⇒データ基盤
- Vポイントの運営会社
  - To C向けには、お得感を提供して
  - To B向けには、セグメント毎の販促基準・分析・リサーチを提供
- 分析基盤の選定
- モダンデータスタックとは？
  - データを使ったプロセスが適切に自走化している状態を生み出すための組織ごとの最適な技術の組み合わせ
  - 最善ではなくて、「最適」
- 過去は、oracle4台とVERTICA
  - Azureに乗り換え（2017年～）
  - Snowflake統合自体（2021年～）
    - 選定理由：アーキテクチャの先進性・移植性の高さとスケーラビリティ・柔軟なコストと自分たちで運用する負担の少なさ
- データエンジニアは8名だけで、全社員600名が使用している
- Tablesuを全社員使用
- データアナリスト100名
- データサイエンティスト20名
- データモデリングを積極的に実施
  - 開発段階からデータエンジニアが参画して、事業部に渡す
- 課題：GoogleのBig Queryでの取り組みが事業側で滞っている⇒エンジニア側で巻き取り予定
- 課題：リアルタイムのポイントAPIログの情報化（現在は、1日1回が前提）
- 課題：非構造化データへの取り組み（検討中）
- データエンジニアがバックエンドだけでなく、フロントエンドに入ってくる世界になっている
- 技術は、手段。
- 課題を常に探し、それをいつどのような手段で解決するかを考える事が重要（いつ：既存システムの契約期間や状況との兼ね合い）
# 「M&Aで拡大し続けるGENDAのデータ活用を促すためのDatabricks権限管理」	株式会社GENDA 馬渡 大樹 氏
- 多様なユーザーが使いやすいDatabricks権限管理をどうするか
  - ユーザー（職種）によって、触りたいものは違うよね
  - 横断的に見たいデータエンジニアとデータアナリストの人の権限が柔軟
- アミューズメント系：カラオケとかゲームセンターとか。グループ会社も多い
- 機械学習もデータ基盤と同じように管理可能
- Unity Catalog
- 各グループ企業ごとにワークスペースを分離
  - リソース競合を回避
  - 見える物を減らして、権限管理をシンプルに
  - 各企業の利用料が可視化しやすい
- 権限の2層構造
  - ワークスペース権限（何ができるか：機能）
  - Unity Catalog権限（何が見えるか：データ）
- グループによる一元管理
  - アカウントレベルグループとグループ会社
- サービスプリンシパルの活用
- 図説が多い・途中退出
